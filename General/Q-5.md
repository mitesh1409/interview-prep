# D in SOLID Principles - DIP - Dependency Inversion Principle

The Dependency Inversion Principle (DIP) states to depend upon abstractions, not concretes.

The principle states:  

High-level modules should not import anything from low-level modules. Both should depend on abstractions (e.g., interfaces).  
Abstractions should not depend on details. Details (concrete implementations) should depend on abstractions.  
By dictating that both high-level and low-level objects must depend on the same abstraction, this design principle inverts the way some people may think about object-oriented programming.

Importance:  

* Loose coupling: Reduces dependencies between modules, making the code more flexible and easier to test.
* Flexibility: Enables changes to implementations without affecting clients.
* Maintainability: Makes code easier to understand and modify.

**Key Terms**  

* **High-level Module:** Contains important business logic or policy. (e.g., A class that handles ordering food.)

* **Low-level Module:** Contains implementation details used to execute a task. (e.g., A class that specifically connects to a SQL database or a specific payment API.)

* **Abstraction (Interface/Abstract Class):** A contract or blueprint that defines the methods but not the implementation.

In essence, DIP "inverts" the typical top-down dependency flow. Instead of the high-level logic being tied to specific low-level implementations, you insert an interface in the middle. The high-level module depends on the interface, and the low-level implementation also implements that interface.

This reduces coupling and makes the code:  

* **Easier to test:** You can swap out the real low-level module with a "mock" implementation for testing.

* **More flexible:** You can change the underlying technology (e.g., switch from MySQL to MongoDB) without altering the core business logic.

## Code Example - Without DIP

Example: Logging System  

```TypeScript
// Low-level Module (Specific Implementation)
class ConsoleLogger {
    log(message: string): void {
        console.log(`[LOG]: ${message}`);
    }
}

// High-level Module
class OrderProcessor {
    // Direct Dependency on Concrete Class!
    constructor(private logger: ConsoleLogger) {
        //
    }

    process(orderId: number): void {
        // Core business logic
        // ...
        this.logger.log(`Order ${orderId} processed successfully.`); 
    }
}

const orderProcessor = new OrderProcessor(new ConsoleLogger());
orderProcessor.process(1001);
```

**Problems:**  

* High level modules and low level modules are tightly coupled.
* Whenever there is a change in low level module, we need to change high level module as well.

**Solution:**  

Apply DIP.

## Code Example - With DIP

Example: Logging System  

```TypeScript
// Abstraction (Interface)
interface ILogger {
    log(message: string): void;
}

// Low-level Module 1 (Concrete Implementation)
class ConsoleLogger implements ILogger {
    log(message: string): void {
        console.log(`[LOG - Console]: ${message}`);
    }
}

// Low-level Module 2 (Concrete Implementation) - Easy to swap in!
class FileLogger implements ILogger {
    log(message: string): void {
        // In a real app, this would write to a file
        console.warn(`[LOG - File]: Writing to file: ${message}`);
    }
}

// High-level Module (Depends on Abstraction - ILogger)
class OrderProcessor {
    // Dependency is injected through the constructor!
    private logger: ILogger; 

    constructor(logger: ILogger) {
        this.logger = logger;
    }

    process(orderId: number): void {
        // Core business logic
        // ...
        this.logger.log(`Order ${orderId} finalized.`); 
    }
}

// Usage: The client decides which concrete logger to use
const consoleProcessor = new OrderProcessor(new ConsoleLogger());
consoleProcessor.process(101); // Uses ConsoleLogger

const fileProcessor = new OrderProcessor(new FileLogger());
fileProcessor.process(102); // Uses FileLogger, and we didn't touch OrderProcessor!
```

Example: Data Persistence  

```TypeScript
// Abstraction (Interface)
interface IUserRepository {
    saveUser(user: { id: number, name: string }): void;
    getUser(id: number): { id: number, name: string } | null;
}

// Low-level Module 1 (Concrete Implementation)
class MongoDBRepository implements IUserRepository {
    saveUser(user: { id: number, name: string }): void {
        console.log(`[MongoDB]: Saving user ${user.name} to MongoDB collection.`);
    }
    getUser(id: number): { id: number, name: string } | null {
        console.log(`[MongoDB]: Fetching user ${id}.`);
        return { id: id, name: "Mongo User" };
    }
}

// High-level Module (Depends on Abstraction - IUserRepository)
class UserService {
    private repository: IUserRepository;

    // Dependency Injection via constructor
    constructor(repository: IUserRepository) {
        this.repository = repository;
    }

    createUser(id: number, name: string): void {
        // Business logic to create user after validating data.
        // ...
        this.repository.saveUser({ id, name });
    }
}

// Using the High-level module with a low-level implementation
const mongoRepo = new MongoDBRepository();
const userService = new UserService(mongoRepo); // Injects the MongoDB dependency

userService.createUser(5, "Alice");
```

**Benefits:**  

* Loose coupling: Reduces dependencies between modules, making the code more flexible and easier to test, easy to mock.
* Flexibility: Enables changes to implementations without affecting clients.
* Maintainability: Makes code easier to understand and modify.

---

**Key Takeaways**  

A high module may depend on one or more lower level modules.  

**Without DIP**  
A high level module depends on concrete low level modules,  
means it directly uses concrete low level modules.  
So whenever there is change in the low level module,  
we need to change high level module as well.  
Here high level modules and low level modules are tightly coupled.  
Dependency flow is Top-Down.  

**With DIP**  
Instead of directly using concrete modules,  
a high level module uses concrete modules via an interface.  
Low level module then implements this interface.  
Even if there is a change in the low level module,  
we don't need to change anything in the high level module.  
Here high level modules and low level modules are loosely coupled.  
Dependency flow is Down-Top (inverted).

## DIP — Dependency Inversion Principle

```markdown
1. High-level modules should not depend on low-level modules.
   Both should depend on abstractions.

2. Abstractions should not depend on details.
   Details (concrete implementations) should depend on abstractions.
```

In plain terms: **your important business logic should not be hardwired to specific tools, databases, or services. It should talk to interfaces — and the concrete implementations plug in from outside.**

## Without DIP

A common example is a notification system. The high-level `OrderService` directly creates and depends on a concrete `EmailNotifier` — it is hardwired to one delivery method.

### TypeScript

```plaintext
// Low-level module — concrete implementation detail
class EmailNotifier {
    // send(message): sends via email
}

// High-level module — directly creates and depends on EmailNotifier
class OrderService {
    // emailNotifier = new EmailNotifier()  <-- hardwired to a concrete class

    // placeOrder(order):
    //   Process order logic
    //   Call emailNotifier.send(...)  <-- cannot swap without editing this class
}
```

```typescript
// Low-level — concrete detail
class EmailNotifier {
  send(message: string): void {
    console.log(`EMAIL: ${message}`);
  }
}

// High-level — directly depends on a concrete low-level class
class OrderService {
  private notifier: EmailNotifier;

  constructor() {
    this.notifier = new EmailNotifier(); // hardwired — DIP violated
  }

  placeOrder(orderId: string): void {
    console.log(`Order ${orderId} placed`);
    this.notifier.send(`Your order ${orderId} has been confirmed`);
    // Want to switch to SMS? You must open and edit this class.
  }
}
```


`OrderService` is a high-level business class — it should not care *how* a notification is delivered. Right now it is forced to.

## With DIP

Introduce an abstraction. Both the high-level `OrderService` and the low-level notifiers depend on it. The concrete implementation is injected from outside — this is called **Dependency Injection**.

### TypeScript

```plaintext
// Abstraction — the contract both sides depend on
interface Notifier {
    // send(message)
}

// Low-level modules — depend on the abstraction, not the other way around
class EmailNotifier implements Notifier {
    // send(message): email logic
}

class SmsNotifier implements Notifier {
    // send(message): SMS logic
}

class PushNotifier implements Notifier {
    // send(message): push notification logic
}

// High-level module — depends on the abstraction, not any concrete class
class OrderService {
    // constructor(notifier: Notifier)  <-- injected from outside

    // placeOrder(order):
    //   Process order logic
    //   Call notifier.send(...)  <-- works with any Notifier, no changes needed
}
```

```typescript
// Abstraction — both sides depend on this
interface Notifier {
  send(message: string): void;
}

// Low-level modules — plug into the abstraction
class EmailNotifier implements Notifier {
  send(message: string): void {
    console.log(`EMAIL: ${message}`);
  }
}

class SmsNotifier implements Notifier {
  send(message: string): void {
    console.log(`SMS: ${message}`);
  }
}

class PushNotifier implements Notifier {
  send(message: string): void {
    console.log(`PUSH: ${message}`);
  }
}

// High-level module — depends only on the abstraction
// The concrete notifier is injected from outside (Dependency Injection)
class OrderService {
  constructor(private notifier: Notifier) {}

  placeOrder(orderId: string): void {
    console.log(`Order ${orderId} placed`);
    this.notifier.send(`Your order ${orderId} has been confirmed`);
  }
}

// Swap notifiers without touching OrderService at all
const emailService = new OrderService(new EmailNotifier());
emailService.placeOrder("001"); // EMAIL: Your order 001 has been confirmed

const smsService = new OrderService(new SmsNotifier());
smsService.placeOrder("002");   // SMS: Your order 002 has been confirmed

const pushService = new OrderService(new PushNotifier());
pushService.placeOrder("003");  // PUSH: Your order 003 has been confirmed

// Testing? Inject a mock — no real emails or SMS sent
class MockNotifier implements Notifier {
  public sentMessages: string[] = [];
  send(message: string): void {
    this.sentMessages.push(message); // capture for assertions
  }
}

const mock = new MockNotifier();
const testService = new OrderService(mock);
testService.placeOrder("TEST-001");
console.log(mock.sentMessages); // ["Your order TEST-001 has been confirmed"]
```

### Python

```plaintext
# Abstraction — the contract both sides depend on
class Notifier(ABC):
    # send(message) — abstract

# Low-level modules — depend on the abstraction, not the other way around
class EmailNotifier(Notifier):
    # send(message): email logic

class SmsNotifier(Notifier):
    # send(message): SMS logic

class PushNotifier(Notifier):
    # send(message): push notification logic

# High-level module — depends on the abstraction, not any concrete class
class OrderService:
    # __init__(notifier: Notifier)  <-- injected from outside

    # place_order(order):
    #   Process order logic
    #   Call notifier.send(...)  <-- works with any Notifier, no changes needed
```

```python
from abc import ABC, abstractmethod


# Abstraction — both sides depend on this
class Notifier(ABC):
    @abstractmethod
    def send(self, message: str) -> None: pass


# Low-level modules — plug into the abstraction
class EmailNotifier(Notifier):
    def send(self, message: str) -> None:
        print(f"EMAIL: {message}")


class SmsNotifier(Notifier):
    def send(self, message: str) -> None:
        print(f"SMS: {message}")


class PushNotifier(Notifier):
    def send(self, message: str) -> None:
        print(f"PUSH: {message}")


# High-level module — depends only on the abstraction
# The concrete notifier is injected from outside (Dependency Injection)
class OrderService:
    def __init__(self, notifier: Notifier):
        self._notifier = notifier

    def place_order(self, order_id: str) -> None:
        print(f"Order {order_id} placed")
        self._notifier.send(f"Your order {order_id} has been confirmed")


# Swap notifiers without touching OrderService at all
email_service = OrderService(EmailNotifier())
email_service.place_order("001")  # EMAIL: Your order 001 has been confirmed

sms_service = OrderService(SmsNotifier())
sms_service.place_order("002")    # SMS: Your order 002 has been confirmed

push_service = OrderService(PushNotifier())
push_service.place_order("003")   # PUSH: Your order 003 has been confirmed


# Testing? Inject a mock — no real emails or SMS sent
class MockNotifier(Notifier):
    def __init__(self):
        self.sent_messages: list[str] = []

    def send(self, message: str) -> None:
        self.sent_messages.append(message)  # capture for assertions


mock = MockNotifier()
test_service = OrderService(mock)
test_service.place_order("TEST-001")
print(mock.sent_messages)  # ["Your order TEST-001 has been confirmed"]
```

## Key takeaways

**`new` inside a constructor is a red flag.** When a high-level class instantiates a concrete low-level class directly, it owns that dependency forever. Injecting it from outside breaks that coupling.

**Dependency Injection is how DIP is practiced.** DIP is the principle — *what* you want. Dependency Injection is the technique — *how* you achieve it. The concrete implementation is passed in (via constructor, method, or a DI container) rather than created internally.

**Testing becomes trivial.** The `MockNotifier` example shows this clearly — you inject a lightweight fake, capture what was sent, and assert against it. No real network calls, no side effects.

**The abstraction is owned by the high-level module.** `Notifier` is defined to serve `OrderService`'s needs — not shaped around what `EmailNotifier` happens to offer. This is the inversion — the high-level module sets the contract, the low-level module conforms to it.

**All five SOLID principles connect here.** `OrderService` has one responsibility (SRP). `Notifier` is the extension point for new delivery methods (OCP). Any `Notifier` implementation substitutes cleanly (LSP). `Notifier` is a focused single-method interface (ISP). And `OrderService` depends on the abstraction, not the detail (DIP).

---

## DIP Checklist

- [ ] High-level classes do not use `new` to instantiate low-level dependencies internally.
- [ ] Dependencies are injected from outside via constructor, method, or a DI container.
- [ ] High-level modules depend on interfaces/abstractions, not concrete classes.
- [ ] Swapping one implementation for another requires zero changes to the high-level module.
- [ ] Tests inject mock/fake implementations without any special setup or monkey-patching.
- [ ] The abstraction is shaped around the high-level module's needs, not the low-level module's convenience.

## SOLID — Complete picture

S — SRP — Single Responsibility Principle
          One class, one reason to change.

O — OCP — Open/Closed Principle
          Open for extension, closed for modification.
          Add new behaviour by writing new classes, not editing existing ones.

L — LSP — Liskov Substitution Principle
          Any subclass must be substitutable for its parent without breaking the program.

I — ISP — Interface Segregation Principle
          No class should be forced to implement methods it does not use.
          Many small interfaces over one fat interface.

D — DIP — Dependency Inversion Principle
          High-level modules depend on abstractions, not concrete low-level modules.
          Inject dependencies from outside.
