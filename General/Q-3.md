# L in SOLID Principles - LSP - Liskov Substitution Principle

The Liskov Substitution Principle (LSP) states that functions that use pointers or references to base classes must be able to use objects of derived classes without knowing it.

An object (such as a class) may be replaced by a sub-object (such as a class that extends the first class) without breaking the program. It is a semantic rather than merely syntactic relation, because it intends to guarantee semantic interoperability of types in a hierarchy, object types in particular.

Replacing a super-class object with a sub-class object won't break the program.

In other words, child should be able to do everything that the parent can do without fail.

> LSP ensures that inheritance preserves behavior, not just structure, so that polymorphism works without surprises.

Importance:  

* Polymorphism: Enables the use of polymorphic behavior, making code more flexible and reusable.
* Reliability: Ensures that subclasses adhere to the contract defined by the superclass.
* Predictability: Guarantees that replacing a superclass object with a subclass object won't break the program.

## Code Example - Without LSP

Suppose we are building a game with some bird characters.  
User can select a bird of his/her choice to play the game.  
User can fly Sparrow & Eagle but in case of Penguin it will not work.  
The end user will then realize there is a bug in the game.  

```TypeScript
// Bird.ts
class Bird {
    fly(): void {
        console.log('A bird can fly...');
    }

    makeSound(): void {
        console.log('A bird can make a sound...');
    }
}

export {
    Bird
};

// Sparrow.ts
import { Bird } from "./Bird.js";

class Sparrow extends Bird {
    fly(): void {
        console.log('A sparrow can fly...');
    }

    makeSound(): void {
        console.log('A sparrow can make a sound...');
    }
}

export {
    Sparrow
};

// Eagle.ts
import { Bird } from "./Bird.js";

class Eagle extends Bird {
    fly(): void {
        console.log('An eagle can fly...');
    }

    makeSound(): void {
        console.log('An eagle can make a sound...');
    }
}

export {
    Eagle
};

// Penguin.ts
import { Bird } from "./Bird.js";

class Penguin extends Bird {
    fly(): void {
        throw Error('A penguin cannot fly');
    }

    makeSound(): void {
        console.log('A penguin can make a sound...');
    }
}

export {
    Penguin
};

// index.ts
import { Bird } from "./Birds/Bird.js";
import { Eagle } from "./Birds/Eagle.js";
import { Penguin } from "./Birds/Penguin.js";
import { Sparrow } from "./Birds/Sparrow.js";

function makeBirdFly(bird: Bird): void {
    bird.fly();
}

makeBirdFly(new Bird());
makeBirdFly(new Sparrow());
makeBirdFly(new Eagle());
makeBirdFly(new Penguin()); // <-- this will break the program as it throws an error - A penguin cannot fly
```

**Problems:**  

Here there is no problem with Bird, Sparrow and Eagle, but Penguin.
Penguin cannot fly and it throws an error.

We can replace Bird object with Sparrow or Eagle object without breaking anything.
But when we replace Bird object with Penguin it fails.

Problem is - Penguin extends Bird but it does not have all the abilities that the Bird has.

**Solution:**  

While inheriting we need to make sure that - a child class must be able to do everything a parent class can do.
If not then we should not inherit/extend from that parent class.

We need to correct class hierarchy here.
Penguin cannot extend Bird since it cannot fly.
We will add one more base class on top of Bird, lets say BaseBird.
Check the code example for more details.

## LSP — Liskov Substitution Principle

The formal definition:

```markdown
If S is a subtype of T, then objects of type T may be replaced with objects
of type S without altering any of the desirable properties of the program.
```

In plain terms: **any child class should be usable wherever its parent class is expected, without breaking the program.** If you have to check what subtype you are dealing with before using it, LSP is likely violated.

---

## Without LSP

The classic example is a `Rectangle` / `Square` problem. A square *is* a rectangle mathematically, so it seems natural to extend `Rectangle` — but it breaks the contract.

### TypeScript

```plaintext
// Base class
class Rectangle {
    // setWidth(w)
    // setHeight(h)
    // area(): width * height
}

// Square extends Rectangle
class Square extends Rectangle {
    // setWidth(w): sets BOTH width and height  <-- breaks the contract
    // setHeight(h): sets BOTH width and height <-- breaks the contract
}

// Consumer expects Rectangle behaviour
function printArea(rectangle: Rectangle) {
    // setWidth(5)
    // setHeight(10)
    // expects area = 50
    // but if a Square is passed: area = 100  <-- surprise!
}
```

```typescript
class Rectangle {
  protected width: number = 0;
  protected height: number = 0;

  setWidth(w: number): void { this.width = w; }
  setHeight(h: number): void { this.height = h; }

  area(): number { return this.width * this.height; }
}

class Square extends Rectangle {
  // Square forces both sides equal — silently breaks the Rectangle contract
  setWidth(w: number): void { this.width = w; this.height = w; }
  setHeight(h: number): void { this.width = h; this.height = h; }
}

function printArea(rect: Rectangle): void {
  rect.setWidth(5);
  rect.setHeight(10);
  console.log(rect.area());
  // Rectangle → 50 ✓
  // Square    → 100 ✗  (height overrode width silently)
}

printArea(new Rectangle()); // 50  — correct
printArea(new Square());    // 100 — unexpected, LSP violated
```

### Python

```plaintext
# Base class
class Rectangle:
    # set_width(w)
    # set_height(h)
    # area(): width * height

# Square extends Rectangle
class Square(Rectangle):
    # set_width(w): sets BOTH width and height  <-- breaks the contract
    # set_height(h): sets BOTH width and height <-- breaks the contract

# Consumer expects Rectangle behaviour
def print_area(rectangle: Rectangle):
    # set_width(5)
    # set_height(10)
    # expects area = 50
    # but if a Square is passed: area = 100  <-- surprise!
```

```python
class Rectangle:
    def __init__(self):
        self._width = 0
        self._height = 0

    def set_width(self, w: int) -> None: self._width = w
    def set_height(self, h: int) -> None: self._height = h

    def area(self) -> int: return self._width * self._height


class Square(Rectangle):
    # Square forces both sides equal — silently breaks the Rectangle contract
    def set_width(self, w: int) -> None: self._width = w; self._height = w
    def set_height(self, h: int) -> None: self._width = h; self._height = h


def print_area(rect: Rectangle) -> None:
    rect.set_width(5)
    rect.set_height(10)
    print(rect.area())
    # Rectangle → 50 ✓
    # Square    → 100 ✗  (height overrode width silently)


print_area(Rectangle())  # 50  — correct
print_area(Square())     # 100 — unexpected, LSP violated
```

## With LSP

The fix is to **not force an inheritance relationship that doesn't hold behaviourally**. Instead, model them as siblings sharing a common abstraction.

### TypeScript

```plaintext
// Shared abstraction — defines only what both truly share
interface Shape {
    // area(): number
}

// Rectangle is its own class — no forced inheritance
class Rectangle implements Shape {
    // constructor(width, height)
    // area(): width * height
}

// Square is its own class — no forced inheritance
class Square implements Shape {
    // constructor(side)
    // area(): side * side
}

// Consumer depends on the abstraction — any Shape works correctly
function printArea(shape: Shape) {
    // shape.area() — behaves correctly for any Shape
}
```

```typescript
// Shared abstraction
interface Shape {
  area(): number;
}

// Each shape owns its own correct behaviour
class Rectangle implements Shape {
  constructor(private width: number, private height: number) {}
  area(): number { return this.width * this.height; }
}

class Square implements Shape {
  constructor(private side: number) {}
  area(): number { return this.side * this.side; }
}

// Works correctly for any Shape — no surprises
function printArea(shape: Shape): void {
  console.log(`Area: ${shape.area()}`);
}

printArea(new Rectangle(5, 10)); // Area: 50  ✓
printArea(new Square(5));        // Area: 25  ✓
```

### Python

```plaintext
# Shared abstraction — defines only what both truly share
class Shape(ABC):
    # area() — abstract

# Rectangle is its own class — no forced inheritance
class Rectangle(Shape):
    # __init__(width, height)
    # area(): width * height

# Square is its own class — no forced inheritance
class Square(Shape):
    # __init__(side)
    # area(): side * side

# Consumer depends on the abstraction — any Shape works correctly
def print_area(shape: Shape):
    # shape.area() — behaves correctly for any Shape
```

```python
from abc import ABC, abstractmethod


# Shared abstraction
class Shape(ABC):
    @abstractmethod
    def area(self) -> int:
        pass


# Each shape owns its own correct behaviour
class Rectangle(Shape):
    def __init__(self, width: int, height: int):
        self._width = width
        self._height = height

    def area(self) -> int:
        return self._width * self._height


class Square(Shape):
    def __init__(self, side: int):
        self._side = side

    def area(self) -> int:
        return self._side * self._side


# Works correctly for any Shape — no surprises
def print_area(shape: Shape) -> None:
    print(f"Area: {shape.area()}")


print_area(Rectangle(5, 10))  # Area: 50  ✓
print_area(Square(5))         # Area: 25  ✓
```

## Key takeaways

**"Is-a" in language does not always mean "is-a" in behaviour.** A square is a rectangle mathematically, but it cannot honour the rectangle's behavioural contract (`setWidth` and `setHeight` are independent). LSP is about *behavioural* substitutability, not taxonomic classification.

**The substitution test** is the practical check: swap every use of the parent with the child — does the program still behave correctly and produce the same expected results? If anything breaks or surprises, LSP is violated.

**Preconditions cannot be strengthened.** A subclass must accept at least what the parent accepts — it cannot demand more from callers.

**Postconditions cannot be weakened.** A subclass must deliver at least what the parent promises — it cannot return less or behave more restrictively.

**The fix is usually flatter hierarchies.** When inheritance feels forced, it often is. Sibling classes implementing a shared interface is almost always cleaner than a deep inheritance chain trying to paper over behavioural differences.

**LSP violations are silent at compile time** in most cases. The code compiles and runs — it just produces wrong results. This makes them particularly sneaky bugs to track down.

---

## LSP Checklist

- [ ] Any subclass can replace its parent class without changing the program's behaviour.
- [ ] The subclass does not override a method in a way that weakens or contradicts the parent's contract.
- [ ] No caller code checks the concrete type (instanceof / typeof) before using the object.
- [ ] The subclass does not throw exceptions the parent does not throw.
- [ ] Preconditions in the subclass are not stricter than in the parent.
- [ ] Postconditions in the subclass are not weaker than in the parent.
- [ ] Inheritance is driven by behavioural compatibility, not just conceptual or taxonomic similarity.
