# O in SOLID Principles - OCP - Open-Closed Principle

The Open–Closed Principle (OCP) states that software entities (classes, modules, functions, etc.) should be open for extension, but closed for modification; that is, such an entity can allow its behaviour to be extended without modifying its source code.

Importance:  

* Extensibility: New features can be added without modifying existing code.
* Stability: Reduces the risk of introducing bugs when making changes.
* Flexibility: Adapts to changing requirements more easily.

References:  

* [Wikipedia > SOLID](https://en.wikipedia.org/wiki/SOLID)

## Code Example - Without OCP

```TypeScript
// PaymentProcessor.ts
class PaymentProcessor {
    amount: number;
    paymentMethod: string;

    constructor(amount: number, paymentMethod: string) {
        this.amount = amount;
        this.paymentMethod = paymentMethod;
    }

    processPayment() {
        try {
            switch (this.paymentMethod) {
                case 'credit-card':
                    console.log('Process payment using Credit Card...');
                    break;
                case 'paypal':
                    console.log('Process payment using Paypal...');
                    break;
                case 'stripe':
                    console.log('Process payment using Stripe...');
                    break;
                default:
                    throw new Error('Invalid payment method');
            }
        } catch (error) {
            console.error('Failed to process payment');
            console.error(error);
        }

        return;
    }
}

export {
    PaymentProcessor
};

// index.ts
import { PaymentProcessor } from "./PaymentProcessor.js";

const paymentAmount = 500;
const paymentMethod = 'credit-card';

const paymentProcessor = new PaymentProcessor(paymentAmount, paymentMethod);

paymentProcessor.processPayment();
```

Here PaymentProcessor class supports 3 payment methods: Credit Card, Paypal & Stripe.  

The class is written in a way that if tomorrow we need to support couple of more payment methods like Debit Card & UPI then we need to modify its code. It is not open for extension.  
Currently PaymentProcessor class is open for modification and closed for extension.  
This violates OCP.

**Problems:**  

* To implement new features, we need to modify the same class
* We need to rewrite/revisit all of its test cases
* Takes longer to implement and roll out new features
* Gradually it starts to become BIG and MESSY as we do more modifications into it

**Solution:**  

Apply OCP.  
A class should be written in a way that it is open for extension and closed for modifications.  

## Code Example - With OCP

```TypeScript
// IPaymentProcessor.ts
interface IPaymentProcessor {
    processPayment(amount: number): void;
}

export type {
    IPaymentProcessor
};

// CreditCardPaymentProcessor.ts
import type { IPaymentProcessor } from "./IPaymentProcessor.js";

class CreditCardPaymentProcessor implements IPaymentProcessor {
    processPayment(amount: number): void {
        console.log(`Process payment of ${amount} using Credit Card...`);
    }
}

export {
    CreditCardPaymentProcessor
};

// DebitCardPaymentProcessor.ts
import type { IPaymentProcessor } from "./IPaymentProcessor.js";

class DebitCardPaymentProcessor implements IPaymentProcessor {
    processPayment(amount: number): void {
        console.log(`Process payment of ${amount} using Debit Card...`);
    }
}

export {
    DebitCardPaymentProcessor
};

// PaypalPaymentProcessor.ts
import type { IPaymentProcessor } from "./IPaymentProcessor.js";

class PaypalPaymentProcessor implements IPaymentProcessor {
    processPayment(amount: number): void {
        console.log(`Process payment of ${amount} using Paypal...`);
    }
}

export {
    PaypalPaymentProcessor
};

// StripePaymentProcessor.ts
import type { IPaymentProcessor } from "./IPaymentProcessor.js";

class StripePaymentProcessor implements IPaymentProcessor {
    processPayment(amount: number): void {
        console.log(`Process payment of ${amount} using Stripe...`);
    }
}

export {
    StripePaymentProcessor
};

// UpiPaymentProcessor.ts
import type { IPaymentProcessor } from "./IPaymentProcessor.js";

class UpiPaymentProcessor implements IPaymentProcessor {
    processPayment(amount: number): void {
        console.log(`Process payment of ${amount} using UPI...`);
    }
}

export {
    UpiPaymentProcessor
};

// PaymentProcessor.ts
import type { IPaymentProcessor } from "./IPaymentProcessor.js";

class PaymentProcessor {
    paymentProcessor: IPaymentProcessor;

    constructor(paymentProcessor: IPaymentProcessor) {
        this.paymentProcessor = paymentProcessor;
    }

    processPayment(amount: number): void {
        this.paymentProcessor.processPayment(amount);
    }
}

export {
    PaymentProcessor
};

// index.ts
import { CreditCardPaymentProcessor } from "./PaymentProcessor/CreditCardPaymentProcessor.js";
import { PaymentProcessor } from "./PaymentProcessor/PaymentProcessor.js";
import { StripePaymentProcessor } from "./PaymentProcessor/StripePaymentProcessor.js";

const amount = 500;

const creditCardPaymentProcessor = new CreditCardPaymentProcessor();
let paymentProcessor = new PaymentProcessor(creditCardPaymentProcessor);
paymentProcessor.processPayment(amount);

const stripePaymentProcessor = new StripePaymentProcessor();
paymentProcessor = new PaymentProcessor(stripePaymentProcessor);
paymentProcessor.processPayment(amount);
```

**Benefits:**  

* To implement new features, we don't need to modify any existing class, we can extend instead (by adding new classes)
* Takes less time to implement new features as we are not touching existing classes/code
* Helps us to keep classes SMALL and CLEAN

> NOTE: This implementation is also called Strategy Design Pattern.

---

**Key Takeaways**  

While writing a class:  

* For a feature if we need to provide multiple options, like payment methods (cards, upi, stripe, paypal etc.), report generation (Text, HTML, PDF etc.), think about defining a common interface and then use it in the main class. That way we make the main class open for extension and close for modification. So later on if we need to implement a new option we just need to write a new class for that without touching existing code in place.
* Check if we have multiple options available to do the same thing, for example,  
  * For travelling - walk, run, bus, car, train, aeroplane etc.
  * For report generation - text, pdf, html etc.
  * For payment - card, upi, net banking, wallet, stripe, paypal etc.

## OCP — Open/Closed Principle

OCP builds naturally on SRP — once each class has one responsibility, OCP ensures you *extend* behaviour by adding new classes rather than editing existing ones.

The core idea: **adding new behaviour should not require touching working, tested code.**

The typical pattern is to define an abstraction (interface or abstract class), then add new behaviour by writing a new implementation — not by opening an existing class and adding another `if/else` or `switch` branch.

---

## Without OCP

### TypeScript

```plaintext
// Process payment
class PaymentProcessor {
    // Check payment method type (if/else or switch)

    // If "credit_card":
    //   Charge via credit card logic

    // If "paypal":
    //   Charge via PayPal logic

    // If "crypto":  <-- adding a new method means editing this class
    //   Charge via crypto logic
}
```

```typescript
class PaymentProcessor {
  process(amount: number, method: string): void {
    if (method === "credit_card") {
      console.log(`Charging $${amount} via Credit Card`);
    } else if (method === "paypal") {
      console.log(`Charging $${amount} via PayPal`);
    } else if (method === "crypto") {
      console.log(`Charging $${amount} via Crypto`);
    }
    // Adding a new payment method = editing this class = violates OCP
  }
}
```

### Python

```plaintext
# Process payment
class PaymentProcessor:
    # Check payment method type (if/elif chain)

    # If "credit_card":
    #   Charge via credit card logic

    # If "paypal":
    #   Charge via PayPal logic

    # If "crypto":  <-- adding a new method means editing this class
    #   Charge via crypto logic
```

```python
class PaymentProcessor:
    def process(self, amount: float, method: str) -> None:
        if method == "credit_card":
            print(f"Charging ${amount} via Credit Card")
        elif method == "paypal":
            print(f"Charging ${amount} via PayPal")
        elif method == "crypto":
            print(f"Charging ${amount} via Crypto")
        # Adding a new payment method = editing this class = violates OCP
```

Every time a new payment method arrives, you crack open `PaymentProcessor`, add another branch, and risk breaking existing behaviour. The class is **never closed**.

## With OCP

### TypeScript

```plaintext
// Define the abstraction — closed for modification
interface PaymentMethod {
    // charge(amount)
}

// Each payment method is its own class — open for extension
class CreditCardPayment implements PaymentMethod {
    // charge(amount): credit card logic
}

class PayPalPayment implements PaymentMethod {
    // charge(amount): PayPal logic
}

class CryptoPayment implements PaymentMethod {
    // charge(amount): crypto logic
}

// Adding a new method tomorrow? Just add a new class. Touch nothing above.
class BankTransferPayment implements PaymentMethod {
    // charge(amount): bank transfer logic
}

// Processor never changes, regardless of how many methods are added
class PaymentProcessor {
    // process(amount, paymentMethod: PaymentMethod)
    //   Call paymentMethod.charge(amount)
}
```

```typescript
// Abstraction — this never changes
interface PaymentMethod {
  charge(amount: number): void;
}

// Existing implementations — closed for modification
class CreditCardPayment implements PaymentMethod {
  charge(amount: number): void {
    console.log(`Charging $${amount} via Credit Card`);
  }
}

class PayPalPayment implements PaymentMethod {
  charge(amount: number): void {
    console.log(`Charging $${amount} via PayPal`);
  }
}

class CryptoPayment implements PaymentMethod {
  charge(amount: number): void {
    console.log(`Charging $${amount} via Crypto`);
  }
}

// New method added tomorrow — zero changes to any class above
class BankTransferPayment implements PaymentMethod {
  charge(amount: number): void {
    console.log(`Charging $${amount} via Bank Transfer`);
  }
}

// Processor is closed for modification — it never needs to change
class PaymentProcessor {
  process(amount: number, paymentMethod: PaymentMethod): void {
    paymentMethod.charge(amount);
  }
}

// Usage
const processor = new PaymentProcessor();
processor.process(100, new CreditCardPayment());
processor.process(200, new PayPalPayment());
processor.process(300, new BankTransferPayment()); // new — nothing edited
```

### Python

```plaintext
# Define the abstraction — closed for modification
class PaymentMethod (ABC):
    # charge(amount) — abstract

# Each payment method is its own class — open for extension
class CreditCardPayment(PaymentMethod):
    # charge(amount): credit card logic

class PayPalPayment(PaymentMethod):
    # charge(amount): PayPal logic

class CryptoPayment(PaymentMethod):
    # charge(amount): crypto logic

# Adding a new method tomorrow? Just add a new class. Touch nothing above.
class BankTransferPayment(PaymentMethod):
    # charge(amount): bank transfer logic

# Processor never changes, regardless of how many methods are added
class PaymentProcessor:
    # process(amount, payment_method: PaymentMethod)
    #   Call payment_method.charge(amount)
```

```python
from abc import ABC, abstractmethod


# Abstraction — this never changes
class PaymentMethod(ABC):
    @abstractmethod
    def charge(self, amount: float) -> None:
        pass


# Existing implementations — closed for modification
class CreditCardPayment(PaymentMethod):
    def charge(self, amount: float) -> None:
        print(f"Charging ${amount} via Credit Card")


class PayPalPayment(PaymentMethod):
    def charge(self, amount: float) -> None:
        print(f"Charging ${amount} via PayPal")


class CryptoPayment(PaymentMethod):
    def charge(self, amount: float) -> None:
        print(f"Charging ${amount} via Crypto")


# New method added tomorrow — zero changes to any class above
class BankTransferPayment(PaymentMethod):
    def charge(self, amount: float) -> None:
        print(f"Charging ${amount} via Bank Transfer")


# Processor is closed for modification — it never needs to change
class PaymentProcessor:
    def process(self, amount: float, payment_method: PaymentMethod) -> None:
        payment_method.charge(amount)


# Usage
processor = PaymentProcessor()
processor.process(100, CreditCardPayment())
processor.process(200, PayPalPayment())
processor.process(300, BankTransferPayment())  # new — nothing edited
```

## Key takeaways

**The `if/else` or `switch` smell** is the most common OCP signal. When adding a new case requires opening an existing class, OCP is being violated. The fix is almost always: extract an abstraction, move each branch into its own class.

**`interface` in TypeScript, `ABC` in Python** are your primary tools. They define the contract (what must be implemented) without prescribing how — leaving each implementation free to do its own thing.

**`PaymentProcessor` never changes** no matter how many payment methods are added. That is the payoff — your stable, tested core is genuinely closed.

**OCP and SRP reinforce each other.** SRP gave each class one responsibility. OCP ensures that adding new responsibilities means writing new classes, not editing existing ones.

**"Open for extension" does not mean open for anything.** The extension point is defined by the abstraction. New behaviour must fit the contract — you are not free to change the shape of `charge()` arbitrarily.

---

## OCP Checklist

- [ ] New behaviour can be added by writing a new class, not editing an existing one.
- [ ] There are no if/else or switch chains that grow every time a new type is added.
- [ ] A shared abstraction (interface or abstract class) defines the extension point.
- [ ] The core processor/orchestrator class does not need to know about concrete implementations.
- [ ] Adding a new variant (e.g. a new payment method) touches exactly one place — the new class itself.
- [ ] Existing classes do not need to be re-tested when new behaviour is introduced.
