# I in SOLID Principles - ISP - Interface Segregation Principle

The interface segregation principle (ISP) states that clients should not be forced to depend upon interfaces that they do not use.

In the field of software engineering, the interface segregation principle (ISP) states that no code should be forced to depend on methods it does not use. ISP splits interfaces that are very large into smaller and more specific ones so that clients will only have to know about the methods that are of interest to them. Such shrunken interfaces are also called role interfaces. ISP is intended to keep a system decoupled and thus easier to refactor, change, and redeploy.

ISP is one of the five SOLID principles of object-oriented design, similar to the "High Cohesion Principle" of GRASP. Beyond object-oriented design, ISP is also a key principle in the design of distributed systems in general and one of the six IDEALS principles for microservice design.

Importance:  

* Decoupling: Reduces dependencies between classes, making the code more modular and maintainable.
* Flexibility: Allows for more targeted implementations of interfaces.
* Avoids unnecessary dependencies: Clients don't have to depend on methods they don't use.

This is like applying SRP to interfaces.

**Problems:**  

There are following problems:  

* Interface IDocumentManager is doing multiple things, we should think about spliting it into multiple interfaces.
* Class BasicCodeEditor does not provide scan, copy and print features but still it has to implement those methods.
* Violation of ISP.

**Solution:**  

Segregate IDocumentManager interface into multiple interfaces to make it simpler.  
That way we will try to apply ISP.

**ISP (Interface Segregation Principle):**  
Focus: interfaces.  
Rule: clients should only depend on the methods they actually use.  
Effect: break large “fat” interfaces into many small/thin, cohesive ones so implementers/consumers aren’t forced to implement or depend on unused methods.  

**SRP (Single Responsibility Principle):**  
Focus: classes/modules.  
Rule: a class/module should have only one reason to change (one responsibility).  
Effect: keeps implementation units small and focused.

## ISP — Interface Segregation Principle

No client should be forced to depend on methods it does not use.

In plain terms: **don't build one fat interface that forces implementors to define methods they have no use for.** Prefer several small, focused interfaces over one large general-purpose one.

---

## Without ISP

A classic example is a multi-function printer/scanner/fax machine. Not every device supports all three — but a fat interface forces every device to implement everything.

### TypeScript

```plaintext
// One fat interface — forces all devices to implement everything
interface Machine {
    // print(document)
    // scan(document)
    // fax(document)
}

// BasicPrinter only prints — but is forced to implement scan and fax
class BasicPrinter implements Machine {
    // print(document): actual print logic

    // scan(document):  does nothing / throws error  <-- forced, unwanted
    // fax(document):   does nothing / throws error  <-- forced, unwanted
}
```

```typescript
// Fat interface — every implementor must define all three
interface Machine {
  print(document: string): void;
  scan(document: string): void;
  fax(document: string): void;
}

class BasicPrinter implements Machine {
  print(document: string): void {
    console.log(`Printing: ${document}`);
  }

  // Forced to implement — but BasicPrinter cannot scan
  scan(document: string): void {
    throw new Error("BasicPrinter cannot scan");
  }

  // Forced to implement — but BasicPrinter cannot fax
  fax(document: string): void {
    throw new Error("BasicPrinter cannot fax");
  }
}
```

### Python

```plaintext

    # scan(document)   — abstract
    # fax(document)    — abstract

class BasicPrinter(Machine):

    # scan(document):  does nothing / raises error  <-- forced, unwanted
    # fax(document):   does nothing / raises error  <-- forced, unwanted
```

```python
from abc import ABC, abstractmethod


# Fat interface — every implementor must define all three
class Machine(ABC):
    @abstractmethod
    def print(self, document: str) -> None: pass

    @abstractmethod
    def scan(self, document: str) -> None: pass

    @abstractmethod
    def fax(self, document: str) -> None: pass


class BasicPrinter(Machine):
    def print(self, document: str) -> None:
        print(f"Printing: {document}")

    # Forced to implement — but BasicPrinter cannot scan
    def scan(self, document: str) -> None:
        raise NotImplementedError("BasicPrinter cannot scan")

    # Forced to implement — but BasicPrinter cannot fax
    def fax(self, document: str) -> None:
        raise NotImplementedError("BasicPrinter cannot fax")
```

The `raise`/`throw` is the dead giveaway — a class implementing an interface it does not fully support is a clear ISP violation.

## With ISP

Split the fat interface into small, focused ones. Each device implements only what it actually supports.

### TypeScript

```plaintext
// Three focused interfaces — each device picks what it supports
interface Printer {
    // print(document)
}

interface Scanner {
    // scan(document)
}

interface Fax {
    // fax(document)
}

// BasicPrinter only prints — implements only Printer
class BasicPrinter implements Printer {
    // print(document): actual print logic
}

// AllInOnePrinter supports everything — implements all three
class AllInOnePrinter implements Printer, Scanner, Fax {
    // print(document): print logic
    // scan(document):  scan logic
    // fax(document):   fax logic
}
```

```typescript
// Small, focused interfaces
interface Printer {
  print(document: string): void;
}

interface Scanner {
  scan(document: string): void;
}

interface Fax {
  fax(document: string): void;
}

// Only implements what it actually supports
class BasicPrinter implements Printer {
  print(document: string): void {
    console.log(`Printing: ${document}`);
  }
}

// Implements only print and scan — no fax
class PrinterScanner implements Printer, Scanner {
  print(document: string): void {
    console.log(`Printing: ${document}`);
  }

  scan(document: string): void {
    console.log(`Scanning: ${document}`);
  }
}

// Implements all three — because it genuinely supports all three
class AllInOnePrinter implements Printer, Scanner, Fax {
  print(document: string): void {
    console.log(`Printing: ${document}`);
  }

  scan(document: string): void {
    console.log(`Scanning: ${document}`);
  }

  fax(document: string): void {
    console.log(`Faxing: ${document}`);
  }
}

// Callers depend only on what they actually need
function printDocument(printer: Printer, document: string): void {
  printer.print(document);
}

function scanDocument(scanner: Scanner, document: string): void {
  scanner.scan(document);
}

printDocument(new BasicPrinter(), "Report.pdf");      // ✓
printDocument(new AllInOnePrinter(), "Report.pdf");   // ✓
scanDocument(new PrinterScanner(), "Report.pdf");     // ✓
```

```python
from abc import ABC, abstractmethod


# Small, focused abstract classes
class Printer(ABC):
    @abstractmethod
    def print(self, document: str) -> None: pass


class Scanner(ABC):
    @abstractmethod
    def scan(self, document: str) -> None: pass


class Fax(ABC):
    @abstractmethod
    def fax(self, document: str) -> None: pass


# Only implements what it actually supports
class BasicPrinter(Printer):
    def print(self, document: str) -> None:
        print(f"Printing: {document}")


# Implements only print and scan — no fax
class PrinterScanner(Printer, Scanner):
    def print(self, document: str) -> None:
        print(f"Printing: {document}")

    def scan(self, document: str) -> None:
        print(f"Scanning: {document}")


# Implements all three — because it genuinely supports all three
class AllInOnePrinter(Printer, Scanner, Fax):
    def print(self, document: str) -> None:
        print(f"Printing: {document}")

    def scan(self, document: str) -> None:
        print(f"Scanning: {document}")

    def fax(self, document: str) -> None:
        print(f"Faxing: {document}")


# Callers depend only on what they actually need
def print_document(printer: Printer, document: str) -> None:
    printer.print(document)


def scan_document(scanner: Scanner, document: str) -> None:
    scanner.scan(document)


print_document(BasicPrinter(), "Report.pdf")     # ✓
print_document(AllInOnePrinter(), "Report.pdf")  # ✓
scan_document(PrinterScanner(), "Report.pdf")    # ✓
```

## Key takeaways

**`NotImplementedError` / `throw new Error` inside an interface method is the loudest ISP smell.** If a class is implementing an interface but throwing on some methods, the interface is too fat.

**Callers should drive interface design.** Ask: *what does this caller actually need?* Design the interface around that — not around what the implementor happens to offer.

**Small interfaces compose freely.** `AllInOnePrinter` implements `Printer`, `Scanner`, and `Fax` without any friction. You lose nothing by splitting — you only gain flexibility.

**ISP and SRP are closely related.** SRP is about a class having one reason to change. ISP is about an interface having one reason to be implemented. A fat interface usually signals that multiple responsibilities have been bundled together.

**ISP applies to function signatures too** — not just interfaces. If a function accepts a large object but only uses two fields from it, that is the same smell at the function level. Pass only what is needed.

---

## ISP Checklist

- [ ] No class implements a method just to leave it empty or throw an error.
- [ ] Each interface is small and focused on one capability.
- [ ] Callers depend only on the interface methods they actually use.
- [ ] Adding a new method to an interface does not force unrelated classes to change.
- [ ] Large interfaces are split into smaller ones that can be composed freely.
- [ ] Function/method parameters accept the smallest type needed, not a large object.

## The core idea

```plaintext
// Each API version is its own interface — clients depend only on their version
interface V1Api {
    // getUser(id)        — available in v1
    // createUser(data)   — available in v1
}

interface V2Api extends V1Api {
    // searchUsers(query) — new in v2
    // deleteUser(id)     — new in v2
}

interface V3Api extends V2Api {
    // bulkCreateUsers(data) — new in v3
}

// Each version has its own service — serves only what that version promises
class V1Service implements V1Api { ... }
class V2Service implements V2Api { ... }
class V3Service implements V3Api { ... }

// Router — routes incoming request to the correct service
class ApiRouter {
    // if version === "v1" → delegate to V1Service
    // if version === "v2" → delegate to V2Service
    // if version === "v3" → delegate to V3Service
}
```

## TypeScript

```typescript
// V1 interface — base capabilities
interface V1Api {
  getUser(id: string): object;
  createUser(data: object): object;
}

// V2 adds new capabilities — v1 clients are unaffected
interface V2Api extends V1Api {
  searchUsers(query: string): object[];
  deleteUser(id: string): void;
}

// V3 adds further capabilities — v1 and v2 clients are unaffected
interface V3Api extends V2Api {
  bulkCreateUsers(data: object[]): object[];
}

// V1 service — implements only v1 contract
class V1Service implements V1Api {
  getUser(id: string): object {
    return { id, name: "Alice" };
  }

  createUser(data: object): object {
    return { id: "123", ...data };
  }
}

// V2 service — implements v1 + v2 contract
class V2Service implements V2Api {
  getUser(id: string): object {
    return { id, name: "Alice", role: "admin" }; // richer response in v2
  }

  createUser(data: object): object {
    return { id: "123", ...data };
  }

  searchUsers(query: string): object[] {
    return [{ id: "1", name: "Alice" }, { id: "2", name: "Bob" }];
  }

  deleteUser(id: string): void {
    console.log(`User ${id} deleted`);
  }
}

// V3 service — implements v1 + v2 + v3 contract
class V3Service implements V3Api {
  getUser(id: string): object {
    return { id, name: "Alice", role: "admin", meta: {} };
  }

  createUser(data: object): object {
    return { id: "123", ...data };
  }

  searchUsers(query: string): object[] {
    return [{ id: "1", name: "Alice" }];
  }

  deleteUser(id: string): void {
    console.log(`User ${id} deleted`);
  }

  bulkCreateUsers(data: object[]): object[] {
    return data.map((d, i) => ({ id: String(i), ...d }));
  }
}

// Router — thin, only routes. Never knows implementation details.
class ApiRouter {
  private routes: Record<string, V1Api> = {
    v1: new V1Service(),
    v2: new V2Service(),
    v3: new V3Service(),
  };

  handle(version: string): V1Api {
    const service = this.routes[version];
    if (!service) throw new Error(`Unsupported API version: ${version}`);
    return service;
  }
}

// Usage
const router = new ApiRouter();

const v1 = router.handle("v1");
console.log(v1.getUser("1"));         // ✓ { id: "1", name: "Alice" }

const v2 = router.handle("v2") as V2Api;
console.log(v2.searchUsers("Alice")); // ✓ only available in v2+

const v3 = router.handle("v3") as V3Api;
console.log(v3.bulkCreateUsers([{ name: "Alice" }])); // ✓ only in v3
```

## Python

```python
from abc import ABC, abstractmethod


# V1 interface — base capabilities
class V1Api(ABC):
    @abstractmethod
    def get_user(self, user_id: str) -> dict: pass

    @abstractmethod
    def create_user(self, data: dict) -> dict: pass


# V2 adds new capabilities — v1 clients are unaffected
class V2Api(V1Api, ABC):
    @abstractmethod
    def search_users(self, query: str) -> list: pass

    @abstractmethod
    def delete_user(self, user_id: str) -> None: pass


# V3 adds further capabilities — v1 and v2 clients are unaffected
class V3Api(V2Api, ABC):
    @abstractmethod
    def bulk_create_users(self, data: list) -> list: pass


# V1 service — implements only v1 contract
class V1Service(V1Api):
    def get_user(self, user_id: str) -> dict:
        return {"id": user_id, "name": "Alice"}

    def create_user(self, data: dict) -> dict:
        return {"id": "123", **data}


# V2 service — implements v1 + v2 contract
class V2Service(V2Api):
    def get_user(self, user_id: str) -> dict:
        return {"id": user_id, "name": "Alice", "role": "admin"}

    def create_user(self, data: dict) -> dict:
        return {"id": "123", **data}

    def search_users(self, query: str) -> list:
        return [{"id": "1", "name": "Alice"}, {"id": "2", "name": "Bob"}]

    def delete_user(self, user_id: str) -> None:
        print(f"User {user_id} deleted")


# V3 service — implements v1 + v2 + v3 contract
class V3Service(V3Api):
    def get_user(self, user_id: str) -> dict:
        return {"id": user_id, "name": "Alice", "role": "admin", "meta": {}}

    def create_user(self, data: dict) -> dict:
        return {"id": "123", **data}

    def search_users(self, query: str) -> list:
        return [{"id": "1", "name": "Alice"}]

    def delete_user(self, user_id: str) -> None:
        print(f"User {user_id} deleted")

    def bulk_create_users(self, data: list) -> list:
        return [{"id": str(i), **d} for i, d in enumerate(data)]


# Router — thin, only routes. Never knows implementation details.
class ApiRouter:
    def __init__(self):
        self._routes: dict[str, V1Api] = {
            "v1": V1Service(),
            "v2": V2Service(),
            "v3": V3Service(),
        }

    def handle(self, version: str) -> V1Api:
        service = self._routes.get(version)
        if not service:
            raise ValueError(f"Unsupported API version: {version}")
        return service


# Usage
router = ApiRouter()

v1 = router.handle("v1")
print(v1.get_user("1"))                          # ✓ {"id": "1", "name": "Alice"}

v2: V2Api = router.handle("v2")
print(v2.search_users("Alice"))                  # ✓ only available in v2+

v3: V3Api = router.handle("v3")
print(v3.bulk_create_users([{"name": "Alice"}])) # ✓ only in v3
```

## Key takeaways for this pattern

**Each version interface is its own contract.** V1 clients depend only on V1Api — they are completely isolated from capabilities added in v2 or v3. This is ISP in action.

**Extending interfaces, not modifying them.** `V2Api extends V1Api` — existing v1 clients and the V1 contract never change. This is also OCP working alongside ISP.

**The router stays thin.** It only maps a version string to a service. No business logic, no branching on what a version can or cannot do — that lives in the service itself.

**Deprecating a version is clean.** Removing v1 means removing `V1Service` from the router's map — one line. Nothing else changes.

**Each version can evolve its responses independently.** Notice `get_user` returns a richer object in v2 and v3 — the same method, but each version delivers what its clients expect without affecting others.
