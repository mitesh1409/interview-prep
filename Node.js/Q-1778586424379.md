# What are child processes? Difference between clusters and child processes.

## Child Processes in Node.js

---

## What Are Child Processes?

Node.js runs in a **single process** by default. A child process is a **separate OS process** spawned by your main Node.js process (the parent).

Since Node.js is single-threaded, it cannot utilize multiple CPU cores on its own. Child processes solve this by letting you:

- Run tasks in **parallel** on separate CPU cores
- Execute **shell commands** or external programs
- **Isolate** heavy or risky work — a child crash doesn't affect the parent
- Communicate back and forth via **IPC (Inter-Process Communication)**

```
OS
├── Parent Process (your Node.js app)    PID: 1000
│       │
│       ├── Child Process 1              PID: 1001  ← separate OS process
│       ├── Child Process 2              PID: 1002  ← separate OS process
│       └── Child Process 3              PID: 1003  ← separate OS process
│       └── Child Process 4              PID: 1004  ← separate OS process
```

Each child process has its own:
- Event Loop
- Call Stack
- Heap (memory)
- PID (Process ID)

## The Four Methods

Node.js `child_process` module provides four ways to create child processes:

### `fork()` — New Node.js Process with IPC

Creates a new **Node.js process** specifically. Sets up an **IPC channel** automatically so parent and child can exchange messages with `.send()` and `.on('message')`.

```javascript
// parent.js
import { fork } from 'child_process';

const child = fork('./child.js');

// Send message to child
child.send({ task: 'processData', payload: [1, 2, 3] });

// Receive message from child
child.on('message', (result) => {
  console.log('Result from child:', result);
});

child.on('exit', (code) => {
  console.log(`Child exited with code ${code}`);
});
```

```javascript
// child.js
process.on('message', (data) => {
  console.log('Child received:', data);

  // Do heavy work here without blocking parent's Event Loop
  const result = data.payload.reduce((a, b) => a + b, 0);

  // Send result back to parent
  process.send({ result });
});
```

---

### `spawn()` — Launch Any Executable, Stream Output

Launches **any OS command or executable**. Output is streamed — good for long-running processes or large output.

```javascript
import { spawn } from 'child_process';

// Run 'ls -la' command
const ls = spawn('ls', ['-la']);

// Output comes as a stream
ls.stdout.on('data', (data) => {
  console.log('Output:', data.toString());
});

ls.stderr.on('data', (data) => {
  console.error('Error:', data.toString());
});

ls.on('close', (code) => {
  console.log(`Process exited with code ${code}`);
});
```

---

### `exec()` — Run Shell Command, Buffer Output

Runs a **shell command** and buffers the entire output — returned via callback when complete. Good for short commands with small output.

```javascript
import { exec } from 'child_process';

exec('git log --oneline -5', (err, stdout, stderr) => {
  if (err) {
    console.error('Error:', err.message);
    return;
  }
  console.log('Git log:', stdout);
});
```

⚠️ Buffers entire output in memory — avoid for commands with large output. Use `spawn()` instead.

---

### `execFile()` — Run a File Directly, No Shell

Like `exec()` but runs a **file directly** without spawning a shell. More efficient and safer — no shell injection risk.

```javascript
import { execFile } from 'child_process';

// Runs the file directly — no shell involved
execFile('node', ['--version'], (err, stdout) => {
  if (err) throw err;
  console.log('Node version:', stdout);
});
```

---

## Comparison of All Four Methods

| Method | Runs | Output | Shell | IPC | Best For |
|---|---|---|---|---|---|
| `fork()` | Node.js file only | Streamed | ❌ | ✅ Built-in | Spawning Node.js workers |
| `spawn()` | Any executable | Streamed | ❌ | ❌ | Long-running commands, large output |
| `exec()` | Shell command | Buffered | ✅ | ❌ | Short shell commands |
| `execFile()` | Any executable | Buffered | ❌ | ❌ | Running files safely, no shell |

## The Relationship — Child Process, Cluster, Worker Threads

```
child_process module — general purpose, create ANY child process
        │
        ├── fork()      ← creates Node.js child process with IPC channel
        ├── spawn()     ← launches any executable, streams I/O
        ├── exec()      ← runs shell command, buffers output
        └── execFile()  ← runs a file directly, no shell

cluster module — specialized, built on child_process.fork()
        │
        └── Specifically designed for HTTP server scaling
            Adds: shared port, built-in load balancing, worker management

worker_threads module — threads, NOT processes
        │
        └── Lightweight, share memory via SharedArrayBuffer
            Designed for CPU-intensive tasks within one process
```

---

## Child Process vs Cluster vs Worker Threads

| | `child_process` | `cluster` | `worker_threads` |
|---|---|---|---|
| Creates | Any child process | Node.js worker processes | Threads inside one process |
| Purpose | General — shell commands, scripts, workers | HTTP server scaling | CPU-intensive tasks |
| Memory | Completely isolated | Completely isolated | Can share via `SharedArrayBuffer` |
| Shared port | ❌ | ✅ Built-in | ❌ |
| Built-in load balancing | ❌ | ✅ | ❌ |
| IPC communication | ✅ `fork()` only | ✅ | ✅ `parentPort.postMessage()` |
| Crash isolation | ✅ | ✅ | ⚠️ Can crash whole process |
| Built on top of | OS process creation | `child_process.fork()` | OS threading |
| Use when | Running scripts, shell commands | Scaling Express server | Image processing, crypto, ML |

---

## When to Use What — Decision Guide

```
Need to scale HTTP server across CPU cores?
        └──▶ cluster (or PM2 in cluster mode)

Need to run a CPU-intensive task without blocking Event Loop?
        └──▶ worker_threads

Need to run a shell command or script?
        └──▶ child_process.exec() or spawn()

Need to spawn a separate Node.js process and communicate with it?
        └──▶ child_process.fork()
```

## The Relationship between Cluster and Child Process

**Cluster and Child Process are two different modules** in Node.js.

```
Child Process (child_process module)
        │
        └── Cluster (cluster module) is built ON TOP of child_process
                    │
                    └── cluster.fork() internally uses child_process.fork()
```

Cluster uses `child_process` under the hood — so yes, cluster workers ARE child processes. But not all child processes are cluster workers.

---

## The Three Ways to Create Child Processes

Node.js has a dedicated `child_process` built-in module with four methods:

| Method | Description |
|---|---|
| `child_process.fork()` | Creates a new Node.js process — can communicate via IPC. This is what `cluster.fork()` uses internally |
| `child_process.spawn()` | Launches any OS command/executable — streams stdio |
| `child_process.exec()` | Runs a shell command — buffers output, returns via callback |
| `child_process.execFile()` | Like `exec()` but runs a file directly without a shell |

---

## Cluster vs Child Process

| | `cluster` | `child_process` |
|---|---|---|
| Purpose | Scale HTTP servers across CPU cores | General purpose — spawn any process |
| What it creates | Multiple Node.js workers sharing a port | Any child process — Node.js or OS commands |
| Load balancing | ✅ Built-in — distributes incoming connections | ❌ You handle it yourself |
| Shared port | ✅ All workers share same port automatically | ❌ Not built in |
| Use case | HTTP server scaling | Running scripts, shell commands, spawning workers |
| Built on top of | `child_process.fork()` | OS-level process creation |

## Quick Example — Child Process Directly

```javascript
import { fork, exec, spawn } from 'child_process';

// fork — create a new Node.js child process
const child = fork('./worker.js');
child.send({ task: 'processData' });
child.on('message', (result) => console.log('Result:', result));

// exec — run a shell command
exec('ls -la', (err, stdout) => console.log(stdout));

// spawn — stream output of a command
const ls = spawn('ls', ['-la']);
ls.stdout.on('data', (data) => console.log(data.toString()));
```

## Summary

- Cluster workers **are** child processes — technically correct
- But `child_process` and `cluster` are **two separate modules** with different purposes
- `cluster` is a **higher level abstraction** built on `child_process.fork()` specifically for HTTP scaling
- Use `cluster` (or PM2) for scaling your Express server
- Use `child_process` directly when you need to run shell commands, scripts, or spawn arbitrary processes
