# SIGINT, SIGTERM, SIGKILL - What are these signals?

## What Are Signals?

Signals are standardized messages sent by the OS (or other processes) to a  
running process to notify it of an event or request an action.  
They originate from the POSIX/Unix standard and are available on all  
Unix-based systems (Linux, macOS). On Windows, support is partial —  
only SIGINT and SIGTERM are supported natively in Node.js on Windows.

---

## The Three Signals

* SIGINT  — Interrupt signal. User presses Ctrl+C in terminal.
* SIGTERM — Termination signal. Sent by process managers, orchestrators,
            or manually via `kill` command to request graceful shutdown.
* SIGKILL — Kill signal. Forces immediate termination by the OS.
            ⚠️ Cannot be caught, handled, or ignored. Ever.

---

## SIGINT (Signal Interrupt)

Triggered when the user presses Ctrl+C in the terminal where  
the process is running.

Common scenarios:
* Developer stops the app locally during development
* A shell script sends Ctrl+C to stop a running process

✅ Can be caught and handled in Node.js.

---

## SIGTERM (Signal Terminate)

It is the standard signal sent by ANY process manager or orchestrator  
that wants to stop your app gracefully.

Common scenarios:  

* Docker — `docker stop` sends SIGTERM, waits 10 seconds, then sends SIGKILL
* Kubernetes — sends SIGTERM during pod shutdown, waits grace period, then SIGKILL
* PM2 — sends SIGTERM when you run `pm2 stop` or `pm2 restart`
* systemd — sends SIGTERM when you run `systemctl stop`
* Heroku — sends SIGTERM during dyno restart or deploy
* Manual — running `kill <pid>` in terminal sends SIGTERM by default

✅ Can be caught and handled in Node.js.

---

## SIGKILL (Signal Kill)

Sent by the OS to forcefully and immediately terminate a process.  
The process has zero opportunity to clean up — it is terminated instantly.

Common scenarios:  

* Docker sends SIGKILL after SIGTERM grace period expires (default 10s)
* Kubernetes sends SIGKILL after terminationGracePeriodSeconds expires
* Developer runs `kill -9 <pid>` in terminal
* OS kills process due to out-of-memory (OOM killer)

❌ CANNOT be caught, handled, or ignored — this is by OS design.
   Any `process.on('SIGKILL')` handler you write will silently do nothing.

---

## Graceful Shutdown — What to Do in Handlers

When SIGINT or SIGTERM is received, the goal is to:  

1. Stop accepting new requests
2. Finish processing all in-flight/ongoing requests
3. Release all resources cleanly
4. Exit with appropriate code

Clean-up operations:  

* Stop accepting new incoming requests (close the HTTP server)
* Wait for ongoing/in-flight requests to complete
* Close database connections (MongoDB, PostgreSQL etc.)
* Close Redis connections / flush pending state updates
* Clear any scheduled jobs or timers (cron jobs, setInterval)
* Flush pending logs to disk or logging service
* Release file handles or any other open resources
* Exit with code 0 (success) for graceful shutdown

---

## Should You Handle All Three?

* SIGINT  — ✅ Yes. Handle it so Ctrl+C triggers graceful shutdown.
* SIGTERM — ✅ Yes. This is the most important one in production.
             Every process manager uses SIGTERM to stop your app.
* SIGKILL — ❌ Cannot be handled. Focus on SIGINT and SIGTERM instead.
             Your goal is to finish cleanup BEFORE SIGKILL arrives.

The key insight: SIGTERM is your warning shot. SIGKILL is the  
execution. Use the SIGTERM window to clean up before SIGKILL arrives.

---

## Code examples

```javascript
// Reusable graceful shutdown function
async function gracefulShutdown(signal) {
    console.log(`${signal} received. Shutting down gracefully...`);

    // Stop accepting new requests
    server.close(async () => {
        // Do cleanup operations...

        // Close database connection
        await mongoose.connection.close();
        console.log('MongoDB connection closed.');

        console.log('Graceful shutdown complete.');
        process.exit(0);
    });

    // Safety net — force exit if graceful shutdown takes too long
    // This handles the case where server.close() hangs due to
    // keep-alive connections that never close
    setTimeout(() => {
        console.error('Graceful shutdown timed out. Forcing exit.');
        process.exit(1);
    }, 10000); // 10 seconds — should finish well before Docker's SIGKILL
}

process.on('SIGINT',  () => gracefulShutdown('SIGINT'));
process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));
```

Remember that the SIGKILL — Cannot Be Handled

```javascript
// ❌ This will NOT work — silently ignored by the OS
process.on('SIGKILL', () => {
  // This callback NEVER runs
  // SIGKILL bypasses the process entirely — OS kills it immediately
  console.log('This will never print');
});
```
