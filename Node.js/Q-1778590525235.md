# What happens if you call `res.end()` twice in an HTTP server?

## Short Answer

The **first `res.end()`** sends the response and closes the connection. The **second `res.end()`** throws an error ‚Äî because you cannot write to a stream that is already finished.

---

## What Error You Get

```
Error [ERR_HTTP_HEADERS_SENT]: Cannot set headers after they are sent to the client
```

or

```
Error: write after end
```

Node.js throws this because the underlying response stream is already in a **closed/finished state** ‚Äî writing to a finished stream is not allowed.

---

## Code Example

```javascript
import http from 'http';

const server = http.createServer((req, res) => {
  res.end('First response'); // ‚úÖ sends response, closes stream

  res.end('Second response'); // ‚ùå throws ERR_HTTP_HEADERS_SENT
});

server.listen(3000);
```

---

## How It Looks in Express

The same happens in Express with `res.send()`, `res.json()`, `res.redirect()` ‚Äî all of them call `res.end()` internally:

```javascript
app.get('/user/:id', async (req, res, next) => {
  const user = await User.findById(req.params.id);

  if (!user) {
    res.status(404).json({ message: 'User not found' }); // ends response
    // ‚öÝÔ∏è missing return ‚Äî code continues executing below!
  }

  // ‚ùå ERR_HTTP_HEADERS_SENT ‚Äî response already sent above
  res.status(200).json({ data: user });
});
```

This is one of the **most common mistakes** in Express ‚Äî forgetting `return` after sending a response.

---

## The Fix ‚Äî Always `return` After Sending a Response

```javascript
app.get('/user/:id', async (req, res, next) => {
  const user = await User.findById(req.params.id);

  if (!user) {
    return res.status(404).json({ message: 'User not found' }); // ‚úÖ returns here
  }

  // Only reached if user was found
  res.status(200).json({ data: user }); // ‚úÖ safe
});
```

---

## How to Protect Against It ‚Äî `res.headersSent`

Node.js provides a built-in flag to check if a response has already been sent:

```javascript
app.get('/example', (req, res) => {
  res.send('Hello');

  // Guard against double response
  if (!res.headersSent) {
    res.send('This will never run');
  }
});
```

`res.headersSent` is `true` once `res.end()` / `res.send()` / `res.json()` has been called. Useful in complex middleware chains where it's hard to track whether a response was already sent.

---

## Common Scenarios Where This Happens

```javascript
// ‚ùå Scenario 1 ‚Äî missing return in if block
if (!user) {
  res.status(404).json({ message: 'Not found' });
  // forgot return ‚Äî falls through to next line
}
res.status(200).json({ data: user });

// ‚ùå Scenario 2 ‚Äî response sent in both try and catch
try {
  const data = await fetchData();
  res.json({ data });    // sent here on success
} catch (err) {
  res.json({ data });    // also sent here ‚Äî double response if fetchData() fails
  next(err);             // and then next(err) may trigger another response
}

// ‚ùå Scenario 3 ‚Äî next() called after res.json()
app.use((req, res, next) => {
  res.json({ message: 'ok' }); // response sent
  next();                       // next middleware also tries to respond
});
```

---

## Key Takeaways

| | Detail |
|---|---|
| What happens | First `res.end()` closes the stream ‚Äî second throws an error |
| Error message | `ERR_HTTP_HEADERS_SENT` or `write after end` |
| Most common cause | Missing `return` after sending a response in Express |
| Prevention | Always `return` after `res.send/json/redirect` |
| Safety check | `res.headersSent` flag tells you if response was already sent |
