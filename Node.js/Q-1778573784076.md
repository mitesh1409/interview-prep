# What happens when you call `require()` multiple times for the same module OR import the same module multiple times?

## How Module Caching Works

The first time a module is loaded, Node.js:

1. **Loads** the file from disk
2. **Executes** the module code top to bottom
3. **Caches** the result in `require.cache` (for CJS) or the ESM module registry

Every subsequent `require()` or `import` of the same module **skips steps 1 and 2** entirely and returns the cached exports directly.

---

## Proving It With Code

```javascript
// counter.js
let count = 0;

count++; // this line runs ONCE no matter how many times module is imported

console.log('Module executed! count =', count);

export default count;
```

```javascript
// app.js
import count1 from './counter.js';
import count2 from './counter.js';
import count3 from './counter.js';

// Output:
// "Module executed! count = 1"  ← printed ONCE, not three times

console.log(count1 === count2); // true — same reference
console.log(count2 === count3); // true — same reference
```

The `console.log` inside `counter.js` prints only **once** — proving the module body executed only once regardless of how many times it was imported.

---

## This is Why the DB Connection Pattern Works

This singleton behaviour is exactly what makes your `database/mongodb.js` pattern reliable:

```javascript
// database/mongodb.js
import mongoose from 'mongoose';

async function connectToDatabase() {
  await mongoose.connect(MONGO_DB_URI);
}

export default connectToDatabase;
```

```javascript
// imported in server.js
import connectToDatabase from './database/mongodb.js';

// imported somewhere else too
import connectToDatabase from './database/mongodb.js';

// Both get the EXACT same function reference from cache
// connectToDatabase() is only ever called once in server.js
// No risk of creating duplicate DB connections from multiple imports
```

Same applies to your Express `app` instance, config objects, logger instance — anything you export is effectively a singleton.

## CJS vs ESM Cache

| | CommonJS (`require`) | ESM (`import`) |
|---|---|---|
| Cache location | `require.cache` object | Internal ESM module registry |
| Cache inspectable? | ✅ Yes — `console.log(require.cache)` | ❌ Not directly accessible |
| Cache key | Resolved file path | Resolved URL/path |
| Singleton guarantee | ✅ Yes | ✅ Yes |

In CJS you can actually inspect and even **delete** from the cache:

```javascript
// CJS only — force a module to reload fresh (clears cache)
delete require.cache[require.resolve('./counter.js')];
const freshModule = require('./counter.js'); // executes module code again
```

This is occasionally used in testing to get a fresh module instance between tests. Not possible with ESM — its cache is not directly manipulable.

---

## One Caveat — Different Paths, Different Cache Entries

The cache key is the **resolved file path**. So if the same file is imported via two different paths that resolve to the same file, they get **two separate cache entries** — breaking the singleton:

```javascript
// ⚠️ These could create TWO cache entries for the same file
import logger from './utils/logger.js';
import logger from '../src/utils/logger.js';
```

This is rare in practice but worth knowing — always use consistent import paths, ideally with path aliases configured.

---

## Module Caching — Singleton Behaviour in Node.js

When a module is first loaded:
1. Loaded from disk
2. Executed (module body runs top to bottom)
3. Result cached — CJS in require.cache, ESM in internal module registry

Subsequent require() or import calls:
- Skip loading and execution entirely
- Return the cached exports directly
- All importers get the same reference — true singleton

This is why:
- DB connection files are safe to import anywhere — connection created once
- Config objects, logger instances, app instance are all singletons
- Module-level side effects (like incrementing a counter) run only once

Caveat:
- Cache key is the resolved file path
- Importing the same file via different relative paths can create
  separate cache entries — always use consistent import paths
