## Observability Stack for Microservices on AWS

Observability has three pillars — **Logs, Metrics, and Traces**. A production system needs all three.

---

## The Three Pillars

| Pillar | Answers | Tool |
|---|---|---|
| **Logs** | What happened? | CloudWatch Logs / OpenSearch |
| **Metrics** | How is the system performing? | CloudWatch Metrics / Prometheus |
| **Traces** | Where did the request slow down or fail? | AWS X-Ray / OpenTelemetry |

## Node.js Application Side

### Structured Logging — Winston

Never use `console.log` in production. Use a structured logger that outputs JSON — machines can parse it, CloudWatch can index it.

```javascript
// utils/logger.js
import winston from 'winston';

const logger = winston.createLogger({
  level: process.env.NODE_ENV === 'production' ? 'info' : 'debug',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()   // structured JSON — CloudWatch can query this
  ),
  transports: [
    new winston.transports.Console(),
    new winston.transports.File({ filename: 'logs/error.log', level: 'error' }),
    new winston.transports.File({ filename: 'logs/combined.log' }),
  ],
});

export default logger;
```

Every log line becomes queryable JSON in CloudWatch:
```json
{
  "timestamp": "2025-05-11T10:00:00Z",
  "level": "error",
  "message": "DB connection failed",
  "service": "subscription-service",
  "requestId": "abc-123"
}
```

### Request Tracing — Correlation IDs

Attach a unique `requestId` to every request so you can trace one request across multiple services:

```javascript
// middlewares/requestId.middleware.js
import { randomUUID } from 'crypto';

const requestId = (req, res, next) => {
  // Use existing ID from upstream service or generate a new one
  req.requestId = req.headers['x-request-id'] || randomUUID();
  res.setHeader('x-request-id', req.requestId);
  next();
};

export default requestId;
```

```javascript
// Use it in every log call
logger.error('Payment failed', {
  requestId: req.requestId,
  userId: req.user.id,
  error: err.message,
});
```

### Distributed Tracing — OpenTelemetry + AWS X-Ray

OpenTelemetry is the vendor-neutral industry standard for tracing. It instruments your app and sends traces to X-Ray:

```bash
npm install @opentelemetry/sdk-node @opentelemetry/auto-instrumentations-node @opentelemetry/exporter-trace-otlp-http
```

```javascript
// tracing.js — must be imported FIRST before anything else in server.js
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';

const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter(),
  instrumentations: [
    getNodeAutoInstrumentations(), // auto-instruments Express, MongoDB, HTTP etc.
  ],
});

sdk.start();
```

Auto-instrumentation captures Express routes, MongoDB queries, HTTP calls — zero manual code needed.

### Health Check Endpoint

Every microservice must expose a health check — used by load balancers and Kubernetes:

```javascript
// routes/health.routes.js
import mongoose from 'mongoose';

app.get('/health', (req, res) => {
  res.status(200).json({
    status: 'ok',
    service: 'subscription-service',
    timestamp: new Date().toISOString(),
    db: mongoose.connection.readyState === 1 ? 'connected' : 'disconnected',
  });
});
```

### Database Side — MongoDB

Enable slow query logging in Mongoose to catch performance issues:

```javascript
// Log any query slower than 100ms
mongoose.set('debug', (collectionName, method, query, doc) => {
  logger.debug('MongoDB query', { collectionName, method, query });
});
```

In **MongoDB Atlas** (managed MongoDB), enable:
- **Performance Advisor** — suggests missing indexes based on real query patterns
- **Real-time Performance Panel** — active operations, slow queries
- **Atlas Alerts** — notify on high CPU, slow queries, connection spikes

## AWS Side

```
Your Services
     │
     ▼
CloudWatch Agent      — collects logs and metrics from EC2/ECS/Lambda
     │
     ├──▶ CloudWatch Logs      — store and search all log streams
     │         │
     │         └──▶ CloudWatch Log Insights  — query logs with SQL-like syntax
     │
     ├──▶ CloudWatch Metrics   — CPU, memory, request count, error rate
     │         │
     │         └──▶ CloudWatch Alarms  — alert when metric crosses threshold
     │                   │
     │                   └──▶ SNS → email/Slack/PagerDuty notification
     │
     └──▶ AWS X-Ray            — distributed traces across all services
               │
               └──▶ X-Ray Service Map  — visual map of all service dependencies
```

Key AWS services to set up:

**CloudWatch Logs** — centralised log storage. Each microservice gets its own log group. Set retention (e.g. 30 days) to control cost.

**CloudWatch Metrics + Alarms** — monitor error rate, latency, CPU. Create alarms that fire when error rate exceeds 5% or p99 latency exceeds 500ms.

**CloudWatch Dashboard** — single pane of glass showing all key metrics across all services in one view.

**AWS X-Ray** — traces requests as they flow across services. The Service Map shows which service is causing latency or errors visually.

**SNS + PagerDuty/OpsGenie** — CloudWatch Alarms → SNS topic → on-call engineer gets paged immediately when something breaks.

## Third Party Tools

| Tool | Purpose | Why |
|---|---|---|
| **Grafana** | Dashboards and visualisation | More powerful than CloudWatch dashboards — widely used |
| **Prometheus** | Metrics collection | Industry standard for metrics in k8s environments |
| **Datadog** | All-in-one observability | Logs + metrics + traces in one platform — very popular in enterprise |
| **Sentry** | Error tracking | Captures exceptions with full stack traces and context — `npm install @sentry/node` |
| **PagerDuty / OpsGenie** | On-call alerting | Routes alerts to the right engineer based on on-call schedule |

For most teams — **Sentry is a must** regardless of what else you choose. It catches and groups errors with full context, far better than searching raw logs.

---

## Recommended Stack by Scale

**Early stage / small team:**
```
Winston (logging) + Sentry (errors) + CloudWatch (AWS native) + X-Ray (tracing)
```

**Mid-scale / growing team:**
```
Above + Grafana + Prometheus + PagerDuty for on-call
```

**Large scale / enterprise:**
```
Above + Datadog (replaces most individual tools with one platform)
```

## One Paragraph Summary

> For observability in a microservices system on AWS, I'd cover all three pillars — logs, metrics, and traces. On the application side, I'd use **Winston** for structured JSON logging with correlation IDs attached to every request for traceability. For distributed tracing, **OpenTelemetry** with the AWS X-Ray exporter auto-instruments Express and MongoDB with zero manual code. Every service exposes a `/health` endpoint. On the AWS side, **CloudWatch** handles centralised log storage and metrics, **X-Ray** provides the service dependency map, and **CloudWatch Alarms** with **SNS** pages the on-call engineer via PagerDuty when thresholds are breached. For error tracking I'd add **Sentry** on every service — it groups exceptions with full context far better than raw logs. For larger teams, **Grafana + Prometheus** or **Datadog** consolidates everything into one observability platform.
