# What is the difference between cluster and worker_threads in Node.js?

Clusters and worker threads have their own Event Loop, Call Stack and Heap.

## Why They Each Have Their Own Everything

**Worker Threads** — same process, multiple threads. Each thread gets its own Event Loop, Call Stack and Heap by design of the threading model.

**Cluster** — these are completely **separate OS processes** (created via `fork()`). Since each is a full independent process, they naturally each have their own Event Loop, Call Stack, and Heap — just like running `node app.js` multiple times in separate terminals.

```
Cluster — Multiple Processes
─────────────────────────────────────────────────────
Master Process
├── Worker Process 1 → [Own Event Loop] [Own Call Stack] [Own Heap]
├── Worker Process 2 → [Own Event Loop] [Own Call Stack] [Own Heap]
├── Worker Process 3 → [Own Event Loop] [Own Call Stack] [Own Heap]
└── Worker Process 4 → [Own Event Loop] [Own Call Stack] [Own Heap]
     (typically one worker per CPU core)

Worker Threads — Multiple Threads Inside ONE Process
─────────────────────────────────────────────────────
Single Process → [Shared Memory available via SharedArrayBuffer]
├── Main Thread     → [Own Event Loop] [Own Call Stack] [Own Heap]
├── Worker Thread 1 → [Own Event Loop] [Own Call Stack] [Own Heap]
├── Worker Thread 2 → [Own Event Loop] [Own Call Stack] [Own Heap]
└── Worker Thread 3 → [Own Event Loop] [Own Call Stack] [Own Heap]
```

---

## Key Differences

| | Cluster | Worker Threads |
|---|---|---|
| What it creates | Separate OS processes | Threads inside one process |
| Memory | Completely isolated — no sharing | Can share memory via `SharedArrayBuffer` |
| Communication | IPC (inter-process communication) — serialized messages | `parentPort.postMessage()` — can transfer raw memory buffers |
| Communication cost | Higher — data must be serialized/deserialized | Lower — can pass memory references directly |
| Crash isolation | ✅ One worker crashes, others unaffected | ⚠️ A thread crash can take down the whole process |
| Startup cost | Higher — full OS process startup | Lower — thread is cheaper than a process |
| CPU cores | Utilizes multiple cores | Utilizes multiple cores |
| Shared server port | ✅ Yes — all workers share same port | ❌ No |
| Use case | Scaling HTTP servers | CPU-intensive tasks |

## When to Use What

### Use Cluster when:
You want to **scale an HTTP server** across all CPU cores to handle more concurrent requests.

```javascript
// cluster.js
import cluster from 'cluster';
import os from 'os';
import app from './app.js';
import { PORT } from './config/env.js';

if (cluster.isPrimary) {
  const numCPUs = os.cpus().length;
  console.log(`Primary process ${process.pid} running`);
  console.log(`Forking ${numCPUs} workers...`);

  // Fork one worker per CPU core
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }

  // Restart a worker if it crashes
  cluster.on('exit', (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} died. Restarting...`);
    cluster.fork();
  });

} else {
  // Each worker runs the Express app on the same port
  app.listen(PORT, () => {
    console.log(`Worker ${process.pid} listening on port ${PORT}`);
  });
}
```

All workers share port `3000` — the OS load balances incoming connections across them automatically.

---

### Use Worker Threads when:
You have a **CPU-intensive task** that would block the Event Loop — image processing, video encoding, complex calculations, ML inference, parsing large files etc.

```javascript
// main.js
import { Worker } from 'worker_threads';

app.post('/process-image', (req, res) => {
  // Offload CPU-heavy work to a worker thread
  // so the main thread's Event Loop stays free for other requests
  const worker = new Worker('./workers/imageProcessor.js', {
    workerData: { imagePath: req.body.imagePath }
  });

  worker.on('message', (result) => {
    res.json({ success: true, data: result });
  });

  worker.on('error', (err) => {
    res.status(500).json({ success: false, message: err.message });
  });
});
```

```javascript
// workers/imageProcessor.js
import { workerData, parentPort } from 'worker_threads';

// This runs on a separate thread — does NOT block the main Event Loop
const result = processImage(workerData.imagePath); // heavy CPU work here

parentPort.postMessage(result); // send result back to main thread
```

## The Core Mental Model

```
Cluster     → scale HORIZONTALLY across CPU cores for more REQUEST THROUGHPUT
                "I need to handle more concurrent HTTP requests"

Worker Threads → scale VERTICALLY for CPU-INTENSIVE WORK within one server
                  "I have a heavy computation that would block my Event Loop"
```

---

## Can You Use Both Together?

Yes — and this is the most powerful pattern for production:

```
Cluster — 8 workers (one per CPU core), each handling HTTP requests
  └── Each worker uses Worker Threads for CPU-heavy tasks
        └── Main thread stays free for the Event Loop
```

```
Incoming Requests
       │
       ▼
Load Balancer
  ├── Cluster Worker 1 (Express) → Worker Thread for heavy task
  ├── Cluster Worker 2 (Express) → Worker Thread for heavy task
  ├── Cluster Worker 3 (Express) → Worker Thread for heavy task
  └── Cluster Worker 4 (Express) → Worker Thread for heavy task
```

In practice though, most Node.js teams use **PM2 in cluster mode** instead of writing raw cluster code — it handles forking, crash recovery, and zero-downtime restarts automatically:

```bash
pm2 start server.js -i max   # -i max = one instance per CPU core
```

This gives you all the benefits of clustering with none of the boilerplate.
