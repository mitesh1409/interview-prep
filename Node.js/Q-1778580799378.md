# Explain Event Loop in Node.js. How it is different from the Browser's Event Loop?

## Event Loop in Node.js

Node.js is single-threaded вҖ” it can only execute one thing at a time. The Event Loop is what allows Node.js to handle thousands of concurrent operations without blocking, by **offloading async work and coming back to it when it's done**.

---

## How It Works вҖ” Simple Mental Model

```
Your Code runs on the Call Stack
        в”Ӯ
        в”ңв”Җв”Җ Synchronous code вҶ’ executes immediately on the stack
        в”Ӯ
        в””в”Җв”Җ Async operation (fs.readFile, setTimeout, HTTP request)
                в”Ӯ
                в”ңв”Җв”Җ Handed off to libuv (OS/thread pool) вҖ” Node moves on
                в”Ӯ
                в””в”Җв”Җ When done вҶ’ callback placed in the appropriate queue
                                        в”Ӯ
                                        в””в”Җв”Җ Event Loop picks it up when
                                            Call Stack is empty
```

---


Everytime the call stack becomes free/empty from the current synchronous code execution,  
Node.js drains two special queues first - **nextTick queue** & **Microtask queue**,  
and then it moves to Event Loop to process Callback Queue (phases of the Event Loop).

- **nextTick queue** вҖ” `process.nextTick()` callbacks вҖ” highest priority
- **Microtask queue** вҖ” resolved Promise callbacks

---

## Execution Priority Order

```javascript
console.log('1 - Synchronous');                        // 1st вҖ” call stack

process.nextTick(() => console.log('2 - nextTick'));   // 3rd вҖ” nextTick queue

Promise.resolve().then(() => console.log('3 - Promise')); // 4th вҖ” microtask queue

setImmediate(() => console.log('4 - setImmediate'));   // 5th вҖ” check phase

setTimeout(() => console.log('5 - setTimeout'), 0);   // 6th вҖ” timers phase

console.log('6 - Synchronous');                        // 2nd вҖ” call stack

// Output:
// 1 - Synchronous
// 6 - Synchronous
// 2 - nextTick
// 3 - Promise
// 4 - setImmediate  (or setTimeout first вҖ” these two can vary)
// 5 - setTimeout
```

Key priority order:
```
Synchronous вҶ’ nextTick вҶ’ Promises вҶ’ setImmediate вҶ’ setTimeout
```

## Node.js vs Browser Event Loop

Both use an Event Loop but they differ in several important ways:

| | Node.js | Browser |
|---|---|---|
| **Environment** | Server вҖ” file system, network, OS | Client вҖ” DOM, user interactions |
| **Phases** | Multiple named phases (timers, poll, check etc.) | Simpler вҖ” task queue + microtask queue |
| **`process.nextTick`** | вң… Exists вҖ” runs before everything else | вқҢ Does not exist |
| **`setImmediate`** | вң… Exists вҖ” runs in check phase | вқҢ Does not exist (non-standard in some browsers) |
| **Microtasks** | Promises drain between every phase | Promises drain after each task |
| **I/O handling** | libuv handles file system, TCP, DNS etc. | Browser Web APIs handle fetch, XHR, timers |
| **Rendering** | вқҢ No rendering step | вң… Has a render step between tasks вҖ” paints UI |
| **Web APIs** | вқҢ No DOM, no `window`, no `document` | вң… Full Web APIs available |
| **Workers** | `worker_threads` module | `Web Workers` API |

---

## The Rendering Difference вҖ” Most Important Distinction

The Browser Event Loop has an extra step Node.js doesn't have:

```
Browser:
Task вҶ’ Microtasks вҶ’ Render (paint UI) вҶ’ Task вҶ’ Microtasks вҶ’ Render ...
                        вҶ‘
              This step doesn't exist in Node.js
              Browser must balance JS execution with UI updates

Node.js:
Phase вҶ’ Microtasks/nextTick вҶ’ Phase вҶ’ Microtasks/nextTick вҶ’ Phase ...
```

This is why in the browser, blocking the Event Loop with heavy JS freezes the UI вҖ” the render step never gets a turn.

---

## Interview One-Liner Summary

> The Event Loop is what makes Node.js non-blocking despite being single-threaded вҖ” it offloads async work to libuv and processes callbacks in a fixed phase order when the Call Stack is empty. Compared to the browser's Event Loop, Node.js has more phases (`timers вҶ’ poll вҶ’ check`), adds `process.nextTick` as the highest priority queue, and has no rendering step вҖ” since there's no UI to paint.

## Question

Between every phase, Node.js drains two special queues first:

* nextTick queue вҖ” `process.nextTick()` callbacks вҖ” highest priority
* Microtask queue вҖ” resolved Promise callbacks

What do we mean by between every phase?
How many callbacks can be executed (max limit) in each phase?

So for example poll phase has 3 i/o callbacks ready in its queue then will it execute all 3 of them and then will move to nextTick queue and then Microtask queue

OR

it will execute 1st callback, at this time callstack is empty and so it will move to nextTick queue and then Microtask queue and then 2nd callback of poll phase, callstack is empty then checks nextTick queue and then microtask queue and so on...

Great question вҖ” this gets into the precise mechanics of how the Event Loop actually works.

---

## The Precise Answer

**Option 2 is correct** вҖ” after **each individual callback**, Node.js checks and drains the nextTick and microtask queues before running the next callback in the same phase.

```
Poll Phase has 3 I/O callbacks ready:

Execute callback 1
        в”Ӯ
        в””в”Җв”Җв–¶ drain nextTick queue completely
        в””в”Җв”Җв–¶ drain microtask queue completely
        
Execute callback 2
        в”Ӯ
        в””в”Җв”Җв–¶ drain nextTick queue completely
        в””в”Җв”Җв–¶ drain microtask queue completely

Execute callback 3
        в”Ӯ
        в””в”Җв”Җв–¶ drain nextTick queue completely
        в””в”Җв”Җв–¶ drain microtask queue completely

Move to next phase (check вҶ’ setImmediate)
```

So "between every phase" is slightly misleading вҖ” more precisely it's **after every single callback execution**, not just at phase boundaries.

---

## Proving It With Code

```javascript
const fs = require('fs');

// Queue up 3 I/O callbacks
fs.readFile('file1.txt', () => {
  console.log('I/O callback 1');
  process.nextTick(() => console.log('  nextTick after callback 1'));
  Promise.resolve().then(() => console.log('  Promise after callback 1'));
});

fs.readFile('file2.txt', () => {
  console.log('I/O callback 2');
  process.nextTick(() => console.log('  nextTick after callback 2'));
  Promise.resolve().then(() => console.log('  Promise after callback 2'));
});

fs.readFile('file3.txt', () => {
  console.log('I/O callback 3');
  process.nextTick(() => console.log('  nextTick after callback 3'));
  Promise.resolve().then(() => console.log('  Promise after callback 3'));
});

// Output:
// I/O callback 1
//   nextTick after callback 1
//   Promise after callback 1
// I/O callback 2
//   nextTick after callback 2
//   Promise after callback 2
// I/O callback 3
//   nextTick after callback 3
//   Promise after callback 3
```

If it were "between phases only", all three I/O callbacks would print first, then all nextTicks, then all Promises. The interleaved output proves nextTick and microtask queues drain after **each individual callback**.

---

## Max Callbacks Per Phase вҖ” Is There a Limit?

Yes вҖ” but it depends on the phase:

**Poll phase** вҖ” no hard limit on number of callbacks. Node.js processes all **currently available** I/O callbacks in the queue, then either:
- Waits for more I/O if nothing else is pending (blocks here)
- Moves on if `setImmediate` is scheduled or timers are ready

**Timers phase** вҖ” processes all `setTimeout`/`setInterval` callbacks whose delay has **expired** at that moment. New timers that expire during this phase are picked up on the **next** tick, not the current one.

**nextTick queue** вҖ” **no limit** вҖ” drains completely every time it's checked. This is actually dangerous:

```javascript
// вҡ пёҸ This starves the Event Loop вҖ” nextTick keeps re-adding itself
function infinite() {
  process.nextTick(infinite); // never lets Event Loop move forward
}
infinite();
```

**Microtask queue (Promises)** вҖ” also drains completely each time it's checked. Same starvation risk:

```javascript
// вҡ пёҸ Also starves the Event Loop
function infinite() {
  Promise.resolve().then(infinite);
}
infinite();
```

---

## Key Takeaways

- nextTick and microtask queues drain after **each individual callback**, not just between phases
- Both queues drain **completely** вҖ” no partial draining
- `process.nextTick` has higher priority than Promises вҖ” always runs first
- Infinite `nextTick` or Promise chains **starve** the Event Loop вҖ” a real production concern
- Each phase processes callbacks that are **currently available** вҖ” new ones added during the phase are generally deferred to the next iteration
