# `this` Binding: Regular Functions vs Arrow Functions

Regular function  

```javascript
function Car(make, model) {
    // 1. 'this' is a new, empty object {}
    this.make = make; // 2. 'this' is used to set properties on the new object
    this.model = model;
    this.year = 2024;
    // 3. The new object is implicitly returned
}

const civic = new Car('Honda', 'Civic');
console.log(`new Binding:`, civic);
```

This works. `this` is bound to the newly created Car Object.

---

Arrow function  

```javascript
const Car = (make, model) => {
    // 1. 'this' is a new, empty object {}
    this.make = make; // 2. 'this' is used to set properties on the new object
    this.model = model;
    this.year = 2024;
    // 3. The new object is implicitly returned
};

const civic = new Car('Honda', 'Civic');
console.log(`new Binding:`, civic);
```

This gives an error - Uncaught TypeError: Car is not a constructor

---

## Regular Function — Works ✅

```javascript
function Car(make, model) {
    this.make = make;
    this.model = model;
    this.year = 2024;
}
const civic = new Car('Honda', 'Civic');
```

When you call a function with `new`, JavaScript does 4 things automatically:

1. Creates a brand new empty object `{}`
2. Binds `this` to that new object inside the function
3. Executes the function body (so `this.make = make` writes to that new object)
4. Returns `this` implicitly (the populated object)

Regular functions are **designed to support this** — they get their own `this` binding at call time, and `new` is what sets it.

---

## Arrow Function — Fails ❌

```javascript
const Car = (make, model) => {
    this.make = make;  // ⚠️ Which 'this' is this?
    ...
};
const civic = new Car('Honda', 'Civic');  // TypeError: Car is not a constructor
```

Arrow functions were intentionally built **without their own `this`**. Instead, they **capture `this` lexically** — meaning they inherit `this` from the surrounding scope where they were *defined*, not where they are *called*.

Because of this fundamental design, JavaScript **forbids** using `new` with arrow functions entirely. The engine doesn't even try to run the body — it throws `TypeError: Car is not a constructor` immediately.

> Think of it this way: `new` needs to inject a fresh `this` into the function, but arrow functions have no slot to receive it.

---

## Side-by-side Comparison

| Feature | Regular Function | Arrow Function |
|---|---|---|
| Own `this` binding | ✅ Yes | ❌ No |
| Can be used with `new` | ✅ Yes | ❌ No |
| `this` determined by | Call site | Lexical (definition) scope |
| `this` in global scope | `window` / `undefined` (strict) | Inherits from outer scope |

---

## Where Arrow Functions *shine* with `this`

The very limitation that breaks constructors is actually a **feature** in callbacks:

```javascript
function Timer() {
    this.seconds = 0;

    // ❌ Regular function — 'this' is lost inside setInterval
    setInterval(function() {
        this.seconds++;  // 'this' is NOT the Timer object here
    }, 1000);

    // ✅ Arrow function — lexically inherits 'this' from Timer
    setInterval(() => {
        this.seconds++;  // 'this' IS the Timer object ✓
    }, 1000);
}
```

---

## The mental model

- **Regular function** → `this` is **dynamic**, decided when the function is *called*
- **Arrow function** → `this` is **static**, decided where the function is *written*
